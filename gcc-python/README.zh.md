# AccessGuard Exchange · 无障碍赏金市场

**一个零信任的智能体经济：让网页变得无障碍——并且只为「可被客观验证」的修复付款。**

> English version: [`README.md`](README.md)

网站所有者（例如政府、公共服务机构）针对某个网页发布「无障碍赏金」。一群相互独立的智能体
会依据 WCAG 2.2 标准**审计**该页面、**竞争**修复其中的缺陷，再由一个中立的**验证者**对修补后的
页面重新测试。只有当修复通过客观的二次审计时，赏金才会在 Solana 上结算——无需信任、无需自证、
无需人工盖章。

## 为什么与 GCC 的使命契合

GCC 支持服务于华语数字公共物品（digital commons）的项目。网页无障碍是一个普遍性问题，但它
**直接影响华语地区的残障公民**：政府与公共服务网站若不可访问，视障、听障、行动不便的使用者
就被挡在必要服务之外。

- **界定并解决公共问题** — 无障碍缺陷是明确的公共问题，本项目不为私利、也不制造公共浪费。
- **立足华语、连接全球** — 引擎具备 **中文（CJK）语言感知**：能识别 `zh` / `zh-Hant` / `zh-Hans`，
  并标记「页面是中文内容却声明了错误 `lang`」这一在华语政府网站上极常见、且会破坏读屏发音的缺陷。
- **可持续增长** — 开源基础设施；公共机构按「已验证的修复」付费，并将部分结算收入**回馈公地**
  （维护免费引擎、为资源不足的华语公民网站提供公益审计），参见 Vitalik 的 revenue–evil 曲线。
- **面向未来** — 新范式：把「付款」绑定到**确定性预言机**（可复现的审计），而非 LLM 的主观判断。

## 与众不同之处

大多数「智能体市场」用 **LLM 充当裁判**来选出赢家——这是一个你必须信任的主观预言机。
AccessGuard 则在一个**客观、确定性的预言机**上结算：同一段 HTML，审计引擎每次都返回相同的
通过/不通过。修复智能体获得报酬，当且仅当**它所声称的那条缺陷确实消失，且没有引入任何新缺陷**。
市场之所以「零信任」，是因为裁判是代码，而不是模型。

## 覆盖的规则（WCAG 2.2 A/AA 的高影响子集）

| 规则 | 准则 |
| --- | --- |
| `img-alt` | 1.1.1 非文本内容 |
| `contrast` | 1.4.3 对比度（最低） |
| `form-label` | 1.3.1 信息与关系 / 4.1.2 名称、角色、值 |
| `control-name` | 4.1.2 名称、角色、值（按钮） |
| `link-name` | 2.4.4 链接目的 |
| `html-lang` | 3.1.1 页面语言（**中文感知**） |
| `doc-title` | 2.4.2 页面标题 |
| `heading-order` | 1.3.1 信息与关系 |

引擎是确定性的、无需网络/链/模型，因此演示不会因为 RPC 或 LLM 抖动而失败。

## 快速开始

```bash
python -m venv .venv && .venv\Scripts\activate   # Windows
pip install -r requirements.txt

python demo.py                                   # 内置英文示例（含 10 处缺陷）
python demo.py samples/inaccessible-zh.html      # 中文政府服务页面示例
python demo.py https://example.gov               # 抓取任意在线页面后运行市场
```

运行中文示例时，你会看到引擎标记出
`html-lang`：**「页面是中文内容却声明 lang="en"」**，随后修复智能体将其修正为 `lang="zh"`，
分数由 `0/100` 提升到 `100/100`。

## 智能体经济的角色

- **需求方**：网站/公共机构智能体，为一个待修复页面出预算。
- **审计智能体**：用确定性引擎给出客观的缺陷清单。
- **修复智能体（多个，相互竞争）**：不同技能分工（对比度专精、表单专精、通用型…）。
- **验证智能体（中立裁判）**：对交付物重新审计——只有「真实改进且无回归」才放款。
- **结算/托管**：Solana 上按「每条已验证缺陷」释放赏金。
